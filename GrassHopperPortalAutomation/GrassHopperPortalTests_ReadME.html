GrassHopper Portal Test Documentation 


Summary: 

This Document walks you through what tool is being used for GrassHopper - Portal Automation, how to install and set it up, structure of the code in detail, how to run tests on local machine. 


Pre-Requisite: 


   Install External Libraries: 


   To Run tests inside this repo, install the following external libraries:


1. Selenium2Library — pip install robotframework-selenium2library

2. RequestsLibrary — pip install robotframework-requests


    Install Browser Drivers:

  

Install Chrome driver and other drivers as needed. Add the location of these browser drivers to your PATH. 


Directory Structure:



GrassHopperPortalAutomation is the root directory for this code. 


DEPENDENCIES.txt:


This file has all the Libraries, Resource files that are needed to Run the GrassHopper Web Tests. 


Environment Directory:


Contains .txt files that represent specific environment configurations where the tests need to run. 


For example, if you want to run tests against Staging, save all of the staging URLs, credentials, test data that are specific to Staging environment in this file. 


You can create more files to represent each environment. 


Resources Directory:


This directory contains 3 files,


Keywords.txt —> GrassHopper specific keywords that we write go in here. 

Locators.txt —> All the Locators on GrassHopper application need to be in this file. 

Variables.txt —> Variables that are constant across various environments need to be in this file. For example, plan names, Pricings and others that are common for GH application can be added to this file. 


TestResults Directory:


This directory is to save the test results after execution. When tests fails, screenshots are saved to this directory as well. 


TestSuite Directory:


This directory contains the test cases. Test Cases could be grouped by features, modules, functionality or any other way. Depending on how test cases are grouped, it could refer to different Resources files if needed. 


Tagging Test cases:


We could use ‘[tags]’ notation to add tags to your test cases. For ex, say your test case is testing PAPI api, you can tag them as API/PAPI. Tags can then be used to run only specific test cases if needed. 


Running Tests:


There are many ways to run test cases from command line. Refer to this page for details. 

http://robotframework.org/robotframework/latest/RobotFrameworkUserGuide.html#all-command-line-options


open terminal and cd into the root directory (GrassHopperPortalAutomation)

$ pybot --variable environment:Staging --outputdir TestResults TestSuite


-- variable is used to pass in variables to the tests from command line. In this case, we are passing the environment name


-- outputdir is used to specify the location of the result files, in this case we are saying save under TestResults directory


TestSuite is the location of the test case files


In this case, all test cases inside TestSuite directory would be run. 


To run only test cases with certain tags, use the -- include attribute in command line as below. 


$ pybot --include papi —variable environment:Staging --outputdir TestResults TestSuite


This will only run test cases that are tagged with the word ‘papi’
